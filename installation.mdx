---
title: "Installation"
description: "Add ruby-saml to your Ruby or Rails application."
---

The `ruby-saml` gem provides the client-side implementation of SAML 2.0 authentication. Add it to your application using Bundler or install it directly via RubyGems.

<Warning>
  Critical vulnerabilities affect `ruby-saml < 1.18.0`, including two authentication bypass vulnerabilities (CVE-2025-25291, CVE-2025-25292, CVE-2025-25293) and additional CVEs through `< 1.18.1`. Always use version `1.18.1` or later.
</Warning>

## Add the gem

<CodeGroup>

```ruby Gemfile (stable)
# Latest stable release
gem 'ruby-saml', '~> 1.18.0'
```

```ruby Gemfile (master)
# Tracking master for bleeding-edge changes
gem 'ruby-saml', :github => 'saml-toolkit/ruby-saml'
```

```bash RubyGems
gem install ruby-saml
```

</CodeGroup>

After updating your `Gemfile`, run:

```bash
bundle install
```

## Setup

<Steps>
  <Step title="Require the gem">
    You can require the entire gem at once, or load only the components your application needs.

    <CodeGroup>

    ```ruby Full gem
    # Loads all ruby-saml components
    require 'onelogin/ruby-saml'
    ```

    ```ruby Individual components
    # Load only what you need
    require 'onelogin/ruby-saml/authrequest'
    require 'onelogin/ruby-saml/response'
    require 'onelogin/ruby-saml/settings'
    require 'onelogin/ruby-saml/metadata'
    ```

    </CodeGroup>

    <Note>
      In a Rails application, `require` calls are unnecessary if you use Bundler — the gem is loaded automatically via `Bundler.require`. Individual component loading is useful in non-Rails Ruby applications where you want to minimize memory overhead.
    </Note>
  </Step>

  <Step title="Configure logging">
    By default, `ruby-saml` emits log messages to `RAILS_DEFAULT_LOGGER` in Rails, and to `STDOUT` outside of Rails. To override this behavior and direct logs to a custom destination, assign a Ruby `Logger` object to the logging singleton:

    ```ruby
    OneLogin::RubySaml::Logging.logger = Logger.new('/var/log/ruby-saml.log')
    ```

    You can use any standard Ruby logger target — a file path, `STDOUT`, `STDERR`, or a custom logger instance.

    <Tip>
      Custom logging is especially useful when troubleshooting SAML response validation errors during initial integration.
    </Tip>
  </Step>

  <Step title="Verify installation">
    Confirm the gem is installed and the version meets the security minimum:

    ```bash
    bundle exec ruby -e "require 'onelogin/ruby-saml'; puts OneLogin::RubySaml::VERSION"
    # => 1.18.1
    ```
  </Step>
</Steps>

## Supported Ruby versions

The following Ruby versions are covered by CI testing:

- **MRI Ruby** 2.1 through 3.4
- **JRuby** 9.1 through 9.4
- **TruffleRuby** (latest)

## Nokogiri dependency

`ruby-saml` depends on [Nokogiri](https://nokogiri.org) for XML parsing. The required version of Nokogiri is determined automatically based on your Ruby version. For most modern Ruby versions (2.6+), Nokogiri `>= 1.13.10` is required.

<Warning>
  **Ruby 1.8.7 only:** Nokogiri dropped support for Ruby 1.8.x in version 1.6. If you are running Ruby 1.8.7, you must pin Nokogiri to a compatible version before installing `ruby-saml`.

  <CodeGroup>

  ```ruby Gemfile
  gem 'nokogiri', '~> 1.5.10'
  ```

  ```bash RubyGems
  gem install nokogiri --version '~> 1.5.10'
  ```

  </CodeGroup>
</Warning>

<Note>
  Some security scanning tools may flag `ruby-saml` as a vulnerability due to its Nokogiri dependency. These are false positives. `ruby-saml` always disables Nokogiri's `DTDLOAD` option and enables its `NONET` option, preventing external entity attacks.
</Note>
